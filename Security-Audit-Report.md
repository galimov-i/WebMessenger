# Отчет по аудиту безопасности приложения WebMessenger

**Дата аудита:** 2026-03-12
**Версия приложения:** текущая (последний коммит 265ff9b)
**Аудитор:** SourceCraft Code Assistant Agent

## 1. Обзор

Приложение WebMessenger представляет собой зашифрованный мессенджер с клиент-серверной архитектурой. Клиентская часть реализована на HTML/CSS/JavaScript, серверная — на Go с использованием SQLite. Сообщения шифруются на стороне клиента с помощью RSA‑OAEP (2048 бит), что обеспечивает сквозное шифрование.

## 2. Методология

Аудит проводился путем статического анализа исходного кода, изучения архитектуры и проверки типовых уязвимостей (OWASP Top 10). Были рассмотрены следующие аспекты:

- Аутентификация и управление сессиями
- Авторизация
- Шифрование и защита данных
- Защита от веб‑атак (XSS, CSRF, инъекции)
- Конфигурация сервера и сети
- Безопасность клиентского кода

## 3. Положительные моменты

- **Сквозное шифрование** — сообщения шифруются на клиенте ключом получателя, сервер не имеет доступа к открытому тексту.
- **Хэширование паролей** — используется bcrypt с cost factor 12.
- **Валидация сложности паролей** — при регистрации оценивается сложность пароля, блокируются слабые пароли, предоставляются рекомендации по генерации.
- **Защита от SQL‑инъекций** — все запросы параметризованы.
- **Экранирование HTML** — клиентский код применяет `escapeHtml` для вывода пользовательского контента.
- **Безопасные заголовки HTTP** — установлены X‑Frame‑Options, X‑Content‑Type‑Options, X‑XSS‑Protection, CSP.
- **Валидация сессий** — проверка срока действия и хранение в БД.

## 4. Выявленные уязвимости и рекомендации

### 4.1. Аутентификация и сессии

| Уязвимость | Уровень риска | Рекомендация |
|------------|---------------|--------------|
| Токен сессии передаётся в URL WebSocket (query‑параметр) | Средний | Избегать передачи токена в URL, так как он может попасть в логи прокси и браузера. Использовать заголовок `Sec‑WebSocket‑Protocol` или аутентификацию через первое сообщение (как сейчас), но без дублирования в URL. |
| Срок жизни сессии — 30 дней | Низкий | Сократить срок до 1–7 дней, реализовать механизм обновления токенов (refresh tokens). |
| Токен хранится в `sessionStorage` | Средний | `sessionStorage` уязвим к XSS. Рассмотреть использование `httpOnly` кук (с флагом `Secure` и `SameSite=Strict`) для хранения токена, если приложение работает в рамках одного домена. |
| Отсутствие rate‑limiting на аутентификацию | Низкий | Добавить ограничение попыток входа (например, 5 попыток в минуту) для защиты от брутфорса. |

### 4.2. Авторизация

| Уязвимость | Уровень риска | Рекомендация |
|------------|---------------|--------------|
| Любой авторизованный пользователь может получить публичный ключ любого другого пользователя | Низкий | Это допустимо, так как публичный ключ предназначен для общего доступа. Однако стоит ограничить список пользователей, если требуется конфиденциальность. |
| Отсутствие проверки, что отправитель не отправляет сообщение самому себе | Низкий | Добавить проверку `sender_id != recipient_id` на стороне сервера. |

### 4.3. Шифрование и хранение данных

| Уязвимость | Уровень риска | Рекомендация |
|------------|---------------|--------------|
| Приватные ключи хранятся в `localStorage` | Высокий | `localStorage` доступен JavaScript, что делает ключи уязвимыми к XSS. Рекомендуется использовать Web Crypto API с непроницаемым хранилищем (например, IndexedDB с защитой от скриптов) или хранить ключи в памяти с возможностью экспорта по запросу пользователя. |
| База данных SQLite не зашифрована | Средний | Файл БД можно зашифровать с помощью SQLCipher или использовать шифрование на уровне файловой системы. Также рекомендуется ограничить права доступа к файлу. |
| Отсутствие шифрования трафика (HTTP) | Высокий | В продакшене необходимо использовать HTTPS/WSS. Для локального тестирования можно использовать самоподписанный сертификат, но в production — TLS от доверенного CA. |

### 4.4. Защита от веб‑атак

| Уязвимость | Уровень риска | Рекомендация |
|------------|---------------|--------------|
| CSP разрешает `unsafe‑inline` для скриптов и стилей | Низкий | Убрать `unsafe‑inline` и вынести inline‑скрипты в отдельные файлы. Это повысит защиту от XSS. |
| CORS разрешает любой origin | Средний | Ограничить `Access‑Control‑Allow‑Origin` конкретными доменами (в production). Для локальной разработки можно оставить, но добавить проверку в production‑конфигурации. |
| Отсутствие CSRF‑токенов | Низкий | Так как API использует Bearer‑токен в заголовке, риск CSRF снижен, но для дополнительной защиты можно добавить CSRF‑токен для state‑changing операций (регистрация, вход, отправка сообщений). |

### 4.5. Конфигурация сервера и сети

| Уязвимость | Уровень риска | Рекомендация |
|------------|---------------|--------------|
| Сервер слушает на всех интерфейсах (`0.0.0.0`) | Низкий | В production ограничить прослушивание только необходимыми интерфейсами (например, localhost или внутренним IP). |
| Отсутствие HTTPS | Высокий | Настроить TLS‑сертификат (Let’s Encrypt) и перенаправлять HTTP на HTTPS. |
| Docker‑контейнер работает от root | Средний | Создать непривилегированного пользователя в Dockerfile и запускать приложение от его имени. |
| Логирование чувствительных данных | Низкий | Исключить из логов токены, ключи и другую чувствительную информацию. |

### 4.6. Клиентская безопасность

| Уязвимость | Уровень риска | Рекомендация |
|------------|---------------|--------------|
| Возможность XSS через вставку HTML в других местах | Низкий | Провести дополнительный аудит на предмет использования `innerHTML` без экранирования. |
| Отсутствие валидации ввода на стороне клиента | Низкий | Добавить валидацию длины и содержания сообщений перед отправкой. |

## 5. Приоритеты исправления

1. **Высокий приоритет**:
   - Переход на HTTPS/WSS.
   - Защита приватных ключей (переход на Web Crypto API или безопасное хранилище).
   - Убрать токен из URL WebSocket.

2. **Средний приоритет**:
   - Шифрование БД SQLite.
   - Ограничение CORS.
   - Запуск Docker‑контейнера от непривилегированного пользователя.
   - Сокращение срока жизни сессии.

3. **Низкий приоритет**:
   - Rate‑limiting.
   - Удаление `unsafe‑inline` из CSP.
   - Добавление CSRF‑токенов.
   - Валидация ввода на клиенте.

## 6. Заключение

Приложение WebMessenger имеет хороший фундамент безопасности: реализовано сквозное шифрование, защита от SQL‑инъекций, корректное хэширование паролей, валидация сложности паролей. Однако есть ряд уязвимостей, связанных с хранением ключей на клиенте, передачей токенов и отсутствием шифрования трафика. Исправление этих проблем повысит общий уровень безопасности и соответствие современным стандартам.

Рекомендуется поэтапно внедрить предложенные улучшения, начиная с наиболее критичных.
